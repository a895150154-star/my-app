## 0. 产品定位（升维版）

### 0.1 业务来源

- **种子来源**：用户在现货交易所担任 AI 产品经理期间的「降本增效」项目
  - 背景：现货交易所运营成本中，散户人工服务（客服+分析师助理）是大头
  - 目标：用 AI 替代部分人工服务工作，降低维护散户的边际成本
- **离职后迭代**：用户将这套方法论从单一公司项目抽离，加入股票场景
- **当前定位**：**金融研究 Agent 的客服化通用方法论产品**（不绑定特定公司、不绑定特定标的）

### 0.2 核心命题

> **把分析师/助理的"答疑 + 解读 + 找内容"工作 AI 化，让散户对手里买的研究服务有掌控感，让公司降本。**

### 0.3 关键判断

| 不重要的事 | 重要的事 |
|---|---|
| 标的是股票还是现货 | **Agent Loop 研究流程本身** |
| 服务的具体公司 | **AI 替代人工的工作模式** |
| 数据接哪个交易所 | **散户的内容掌控感设计** |

**所以产品演示要"股票 + 现货并存"，体现方法论的通用性。**

## 1. 切片一 · 散户原本怎么问

### 1.1 提问渠道（信心：🟡 中，待用户确认）

推演：微信群 / 企业微信 / App 内私信 这三种最常见。

### 1.2 典型问题分类

| 类型 | 示例 | 占比推演 | 信心 |
|---|---|---|---|
| **A. 行情类** | "XX 今天跌了，我该止损吗？" | 高频 | 🟢 高 |
| **B. 持仓类** | "我手上的 MG2406，现在该加仓还是减仓？" | 高频 | 🟢 高 |
| **C. 内容找回类** | "老师上周三发的周报在哪？我翻不到" | 中频 | 🟡 中 |
| **D. 解读类** | "你们发的研报里写'供给收缩'，啥意思？" | 中频 | 🟡 中 |
| **E. 跟单类** | "老师之前说看多镁合金，现在还看多吗？" | 高频 | 🟢 高 |
| **F. 操作类** | "止损位设在哪？要不要全部清仓？" | 高频 | 🟢 高（但**合规高危**） |
| **G. 情绪安抚** | "套住了好慌，怎么办？" | 中低频 | 🟡 中 |
| **H. 闲聊** | "老师吃饭了吗" | 低价值 | 🔴 真实场景常见，但 AI 不应接 |

### 1.3 设计要点

- AI 必须能识别 F/G/H 类，**不能装懂去答 F**，不应浪费算力答 H
- C/D 是 AI 最容易吃下来的（高确定性、低风险）
- A/B/E 是 AI 主战场（高频 + 中等风险，需要置信度兜底）

## 2. 切片二 · 分析师助理原本怎么答

### 2.1 推演的助理工作流（信心：🔴 低，强烈建议用户核实）

```
散户发问 (群/私信)
   ↓
助理看到 (可能延迟几小时)
   ↓
助理判断分流：
  ├─ 简单问题 (查日历/找研报/复述常识) → 助理自己答
  ├─ 中等问题 (个性化操作建议)         → 助理转给分析师，等回复后转达
  └─ 复杂问题 (深度研究)                → 让分析师直接进群语音 / 发文档
   ↓
散户收到回复 (可能等了半天)
```

### 2.2 推演的助理工作时间占比

- 60% · 在不同群/私信里**重复回答相同问题**
- 20% · **找历史内容**（"上次老师说啥来着"）
- 15% · **传话**（散户问 ↔ 分析师答 中间转）
- 5% · 真正有价值的服务

### 2.3 推演的人力配比

- 每个分析师配 1-2 个助理
- 一个助理服务 50-200 个散户

> ⚠️ 上述配比信心很低，待用户核实。这关系到 AI 替代后能省多少人。

## 3. 切片三 · 真实痛点排序

### 3.1 推演的 3 大痛点

| 排序 | 痛点 | 谁的痛 | 推演逻辑 |
|---|---|---|---|
| 🥇 #1 | **重复答疑成本爆炸** | 公司付助理工资 | 散户问的 60% 是同一批问题，助理人工答 N 遍 |
| 🥈 #2 | **响应慢导致续费率低** | 续费决策者 | 散户等几小时才有回复，体验差 → 不续费 |
| 🥉 #3 | **分析师被打扰，产能受损** | 分析师本人 | 助理转问题给分析师，分析师一天被打断 N 次，写不出深度研报 |

### 3.2 商业化角度的主指标

老板/领导最在乎 **#1（人力成本数字最直白，可量化、可汇报）**。

但 #2 是**长期最大杠杆** —— 续费率每提升 5%，营收复利效应远超人力成本节省。

**FIN AI 应该同时打 #1（看得见的省钱）+ #2（看不见的赚钱）。**

## 4. 切片四 · AI 介入方案

### 4.1 推演的 AI 处理流程

```
散户发问 (微信群/企微/App)
   ↓
AI 第一道处理（毫秒级）
   ├─ 命中常见问题库 (FAQ + 历史问答相似度匹配) → AI 直接答
   ├─ 命中研报库 (找到对应分析师内容)         → AI 摘要 + 附原文链接
   ├─ 涉及个性化持仓                          → AI 给「建议草稿」转助理审核
   └─ 涉及合规红区 (买卖建议/目标价/保赚)     → AI 不答，强制转人工
   ↓
散户秒回 / 人工兜底
```

### 4.2 关键设计 · 4 条死规则

1. **AI 输出必须标"分析师 XX 在 XX 研报中提到"**
   - 保留分析师 IP 署名感
   - 让散户感觉是"分析师的内容被 AI 帮忙整理"，不是"AI 替代了分析师"

2. **AI 涉及买卖建议必须降级为"研判信息"**
   - 不说"建议买"，说"分析师认为偏多，您的持仓比例可参考……"
   - 主语永远是分析师，AI 是转述者

3. **AI 答不了的必须能转人工，不能装懂**
   - 置信度低于阈值 → 自动转助理 → 助理审过 → 人工回复
   - **置信度本身就是商业化关键指标**：高置信度比例越高，省的人力越多

4. **散户问的所有问题 + AI 答的所有内容必须留痕**
   - 留给运营复盘（哪些问题高频 → 优化 FAQ 库）
   - 留给合规审计（监管要求）
   - 留给知识库回写（AI 的好答案沉淀为新 FAQ）

## 6. 完整流程图

```
                        【散户】
                          │
            ① 在微信群/企微/App 提问
                          │
                          ▼
                  ┌───────────────┐
                  │   FIN AI 引擎  │
                  └───────────────┘
                          │
              ② 5 步 Agent Loop 处理
                ├─ 拉行情 (实时/历史)
                ├─ 检索研报库 (找到对应分析师内容)
                ├─ 综合推理 (多步思考)
                ├─ 事实校验 (相似度 + 置信度评分)
                └─ 生成结构化输出
                          │
                          ▼
                  ┌──────────────┐
                  │  路由决策器   │
                  └──────────────┘
                  │      │       │
        ③-A 直接给       ③-B 转    ③-C 转分析师
        散户答案         助理审    本人处理
        (高置信度)       (中等)    (复杂/合规高危)
                  │      │       │
                  ▼      ▼       ▼
              【散户秒回】     【人工兜底】
                          │
            ④ 所有对话留痕 → 知识库回写 → 持续优化
```
