# 代码评审流程

对当前分支的变更进行评审：$ARGUMENTS

## Step 1：变更范围确认
- 运行 `git diff` 查看所有变更
- 列出修改的文件及改动行数
- 确认变更范围与 spec.md 一致，不存在范围蔓延

## Step 2：代码质量检查
逐文件检查：
- 是否符合 coding-style.md 规范
- 是否引入安全漏洞（注入、XSS、硬编码密钥等）
- 是否有未处理的错误路径
- 命名是否清晰，逻辑是否可读

## Step 3：业务逻辑检查
- 金融数据字段类型是否正确（整数/分，非浮点）
- 交易相关操作是否有人工确认环节
- Agent 输出是否有置信度标注和数据源引用

## Step 4：输出评审报告
在 `changes/` 对应目录创建 `code_review_v{N}.md`：

评审意见分三级：
- **MUST FIX**：必须修复才能合并（安全漏洞、逻辑错误、金融合规问题）
- **SHOULD FIX**：强烈建议修复（代码质量、可维护性）
- **INFO**：建议了解（优化建议、风格偏好）

每条意见包含：文件路径 + 行号 + 问题描述 + 修改建议
